# 仕様書上の不明点と採用解釈

## ISSUE-01 パス表記の不一致
- 観測: 問題文では `.copilot-spec/` とされているが、実際のリポジトリには `copilot-spec/` が存在する。
- 採用解釈: 実在パス `copilot-spec/` を正本とみなし、読了・参照・監査はすべてこちらで行う。
- 理由: 実ファイル参照不能を回避し、停止禁止原則を守るため。

## ISSUE-02 manifest の呼称揺れ
- 観測: `02_meta_files.md` では `_meta/manifest.md`、問題文では `_meta/manifest.md` と `manifest.md` の両方が現れる。
- 採用解釈: `_meta/manifest.md` を唯一の実体とし、Phase 11 の README から導線を張る。
- 理由: メタ情報の一元化を優先するため。

## ISSUE-03 骨格段階とPhase 0調査の順序
- 観測: 08では骨格確定を先行、04ではPhase 0調査を先頭に置く。
- 採用解釈: まず `_meta/` と命名・時代・地方・事件枠の骨格を固定し、その直後にPhase 0調査を大量実施して下流へ反映する。
- 理由: 固有名詞乱造防止と検索知見の反映先明確化を両立するため。

## ISSUE-04 検索40回の単位
- 観測: 08と09で「Web検索40回以上」とあるが、検索1回の粒度は未定義。
- 採用解釈: 検索クエリ1件を1回と数え、採用・不採用を含め `search_log.md` に全件記録する。
- 理由: 最も厳密で監査しやすい運用だから。
